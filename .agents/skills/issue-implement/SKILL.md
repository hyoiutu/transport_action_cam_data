# AI Issue実装スキル (issue-implement)

このスキルは、AIエージェントがGitHub Issueの内容をもとに、人間の都度の指示を介さずに実装・修正を進めるための指示書です。

## 🎯 起動条件

ユーザーから「Issue #<番号>を実装して」「GitHub Issueを対応して」等、GitHub Issueに基づいて実装・修正を進める指示があった場合、または本スキルが明示的に呼び出された場合に起動してください。

---

## 📋 実行プロセス（手順）

### 1. Issue一覧・詳細の取得
- GitHub MCPサーバー（`list_issues`等）を用いて、対象リポジトリのIssue一覧（特にOPEN状態のもの）を取得します。
- ユーザーが対象Issue番号を既に明示している場合も、内容を正確に把握するため該当Issueの詳細を取得します。

### 2. ユーザーへの確認（AskUserQuestion）
- 実装（および後述のブランチ作成）を開始する前に、**必ず**`AskUserQuestion`等を用いて以下の3点をユーザーに確認してください。この確認を省略して実装やブランチ作成を進めてはいけません。
  1. **対象Issue**: どのGitHub Issue（番号）を対象に実装を進めるか。
  2. **ブランチ名**: ユーザーの希望があればそれを採用し、希望がなければ[branch_rules.md](../../../branch_rules.md)の命名規則（`<type>/issue-<Issue番号>-<内容>`）に従いAIが提案します。
  3. **派生元ブランチ**: デフォルトは現在のブランチです。ユーザーが別ブランチを希望すればそちらを採用します。

### 3. ブランチ運用規約の確認とブランチの用意
- [branch_rules.md](../../../branch_rules.md)を読み込み、「1 Issue = 1 Branch」の原則やブランチ命名規則、プッシュ禁止ルールを再確認します。
- 対象Issue用のブランチが既に存在する場合は新規作成せず、そのブランチをチェックアウトして再利用します（`git checkout <ブランチ名>`）。
- 存在しない場合は、手順2で確認した派生元ブランチから新規作成します（`git checkout -b <ブランチ名> <派生元ブランチ>`）。
- 派生元ブランチに未コミットの変更が残っている場合は、ブランチを切る前にユーザーに状況を伝え、確認を得ます。

### 4. 実装の実施（既存プロジェクトルールの遵守）
- 対象Issueの内容に基づき実装を行います。その際、AGENTS.md / rules.md / test_rules.md に定義された既存ルールを必ず遵守してください。
  - TDD（Red-Green-Refactor）に従って実装すること。
  - 実装・README.md・specs/以下の仕様書のいずれかに修正が生じた場合、3者間の乖離がないか確認し、乖離があれば連動して修正すること。
  - 上記の連動修正が発生した場合、CHANGELOG.mdに変更内容（何が変わり、実装・README.md・仕様書がそれぞれどう修正されたか）を記録すること。
  - コード規約はrules.mdに従うこと。
  - 単体テスト・E2Eテストの作成・実行はtest_rules.mdに従うこと。

### 5. コミット
- 実装が完了し、コミット可能な差分が生じた場合は、必ず[auto-commitスキル](../auto-commit/SKILL.md)（[commit_rules.md](../../../commit_rules.md)）の手順を経由してコミットを行います。**本スキルおよびAIエージェントが直接`git commit`を実行してはいけません。**

### 6. 🚫【警告】プッシュの絶対禁止
- **本スキル、あるいはAI自身の手によって`git push`を実行することは絶対に禁止されています。** リモートリポジトリへの送信は必ずユーザー（人間）自身が行う必要があります。
